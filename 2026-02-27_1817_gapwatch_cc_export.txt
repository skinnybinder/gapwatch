❯ why pin it? 

● The toolchain directive locks the exact version used to build, so anyone
  cloning the repo — and CI — builds with the same toolchain rather than
  whatever happens to be installed. Without it, go 1.23 only sets a minimum; the
   actual toolchain could be 1.23.1, 1.24, anything.

  For a security fix specifically, pinning makes the remediation auditable: the
  fix is go1.23.10, not "1.23 or later."

  That said, it does mean the toolchain gets downloaded on first build if the
  local version differs. If you'd rather not have that friction, dropping
  toolchain go1.23.10 and keeping only go 1.23 is reasonable — the vulnerability
   is still fixed as long as anyone building uses 1.23.10+.

● Go has the same problem Python does with version management — it just hides it
   until it bites you.

  The short version:

  - go 1.23 in go.mod = "don't build this with anything older than 1.23"
  - toolchain go1.23.10 = "use exactly this version, download it if needed"

  The vulnerability was in Go 1.21. Bumping to 1.23 is the fix. The toolchain
  line is optional housekeeping on top.

  If you want to keep it simple, just commit what's there. It works and it's
  correct.

❯ so whats the roadmap to updating go versions? 

● For this project:

  1. Check for a new Go release — go.dev/dl or govulncheck will tell you when
  something's broken
  2. Update go.mod — bump the go line, update toolchain if pinned
  3. go mod tidy — updates go.sum, resolves any compat issues
  4. Run tests — go test ./cmd/gapwatch/ -v
  5. Run govulncheck — confirm nothing new landed

  That's it. No equivalent of pip freeze rot or virtual env conflicts. Go's
  stdlib is versioned with the toolchain, so bumping the toolchain is the whole
  job.

  Cadence: Go releases two minor versions per year (Feb and Aug). Each is
  supported for one year. You'd realistically do this once or twice a year.
